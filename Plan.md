# План автоматизации тестирования

## Автоматизируемые сценарии

Автоматизация выполняется для формы «Купить» — оплата тура дебетовой картой.

1. Успешная оплата APPROVED-картой `1111 2222 3333 4444`.
2. Отклонение оплаты DECLINED-картой `5555 6666 7777 8888`.
3. Отклонение неизвестной карты.
4. Валидация истёкшего года действия карты.
5. Валидация некорректного месяца, например `13`.
6. Валидация обязательности поля «Владелец».
7. Проверка, что в БД сохраняется статус операции и создаётся запись заказа.
8. Проверка, что данные банковских карт не сохраняются в БД.

## Инструменты

- Java 11 — язык реализации автотестов.
- Gradle — сборка проекта и запуск тестов.
- JUnit 5 — тестовый фреймворк.
- Selenide — UI-автоматизация браузера.
- JDBC — прямые проверки записей в БД.
- MySQL/PostgreSQL — поддерживаемые СУБД приложения.
- Docker Compose — быстрый запуск тестовой БД.
- Allure — отчётность по результатам автотестов.
- GitHub Issues — фиксация найденных дефектов.

## Риски

- Нестабильность внешнего банковского симулятора.
- Отличия поведения приложения при MySQL и PostgreSQL.
- Изменение локаторов интерфейса.
- Накопление тестовых данных в БД при падении тестов.
- Возможные ошибки в приложении, из-за которых часть тестов должна падать.

## Оценка времени

- Изучение приложения и подготовка окружения: 3–5 часов.
- Проектирование тест-кейсов: 2–3 часа.
- Ручное тестирование и оформление issue: 4–6 часов.
- Автоматизация UI-тестов: 6–10 часов.
- Реализация проверок БД: 3–5 часов.
- Подготовка отчётов и README: 2–4 часа.

Итого: 20–33 часа с учётом рисков.

## План сдачи

1. День 1–3: подготовить Plan.md и тестовые сценарии.
2. День 4–5: провести ручное тестирование, оформить issue.
3. День 6–8: реализовать автотесты и проверки БД.
4. День 9: прогнать тесты, сформировать Allure/Gradle отчёты.
5. День 10: подготовить Report.md, Summary.md и README.md.
